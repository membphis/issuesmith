# Code Review

## 概述

在问题蔓延之前做 review。Review 的是产物（代码），不是思路（聊天记录）。

**核心原则：** Review 早于合并，每次提交后、合并前。

## 何时 Review

**必须 review：**
- 实现阶段每完成一个主要任务后
- `/ism:finish` 的提交后自检阶段
- 合并到 main 之前

**建议 review：**
- 卡住时（新鲜视角）
- 重构之前（基线检查）
- 修复复杂 bug 后

## Review 流程

**1. 确定变更范围：**

```bash
BASE_SHA=$(git merge-base origin/main HEAD)
HEAD_SHA=$(git rev-parse HEAD)

git diff --stat $BASE_SHA..$HEAD_SHA
git diff $BASE_SHA..$HEAD_SHA
```

**2. 按以下维度审查变更：**

### 计划对齐

以下是你应该检查的维度：

- **Issue 对齐：** 实现是否匹配 Issue 的 Goal 和 Acceptance Criteria？
- **偏差判断：** 偏离的部分是合理的改进，还是成问题的偏离？
- **完整性：** 所有计划的功能都实现了吗？Non-goals 范围被遵守了吗？

### 代码质量

- 关注点分离清晰吗？
- 错误处理完善吗？
- 类型安全（在适用语言中）？
- DRY 但不过度抽象？
- 边界情况处理了吗？

### 架构

- 设计决策合理吗？
- 可扩展性和性能合理吗？
- 安全问题？
- 与周边代码整合顺畅吗？

### 测试

- 测试验证的是真实行为，不是 mock？
- 边界情况覆盖了吗？
- 关键路径有集成测试吗？
- 所有测试通过？

### 生产就绪

- Schema 变更时有迁移方案吗？
- 向后兼容性考虑了吗？
- 文档完整吗？
- 有明显 bug 吗？

**3. 分级输出：**

按实际严重程度分级。别把所有问题都标 Critical。在列出问题前先肯定做得好的地方 — 准确的肯定让反馈更可信。

## 输出格式

```
### 优点
[做得好的地方。要具体。]

### 问题

#### Critical（必须修复）
[Bug、安全问题、数据丢失风险、功能缺失]

#### Important（应该修复）
[架构问题、遗漏功能、错误处理不足、测试缺口]

#### Minor（可选修复）
[代码风格、优化机会、文档完善]

对每个问题提供：
- 文件:行号
- 什么问题
- 为什么重要
- 如何修复（如果不明显）

### 建议
[代码质量、架构或流程的改进建议]

### 评估

**可以合并？** [是 | 否 | 修复后]

**理由：** [1-2 句技术评估]
```

## 处理 Review 结果

- **Critical** — 立即修复再合并
- **Important** — 在进入下一步之前修复
- **Minor** — 记录起来，后续处理
- **Review 有误** — 用技术推理反驳，展示代码/测试证明

## 示例

```
### 优点
- 数据库 schema 干净，迁移文件完整（db.ts:15-42）
- 测试覆盖全面（18 个测试，含所有边界情况）
- 错误处理稳健，有 fallback 机制（summarizer.ts:85-92）

### 问题

#### Important
1. CLI 包装器缺少帮助文本
   - 文件: index-conversations.ts:1-31
   - 问题: 没有 --help 标志，用户无法发现 --concurrency 选项
   - 修复: 添加 --help case，展示使用示例

2. 日期校验缺失
   - 文件: search.ts:25-27
   - 问题: 无效日期静默返回空结果
   - 修复: 验证 ISO 格式，无效时抛出带示例的错误

#### Minor
1. 进度指示器
   - 文件: indexer.ts:130
   - 问题: 长时间操作没有 "X / Y" 计数器
   - 影响: 用户不知道要等多久

### 建议
- 为长时间运行的操作添加进度报告
- 考虑为被排除的项目添加配置文件

### 评估

**可以合并：修复后**

**理由：** 核心实现扎实，架构和测试都不错。Important 问题（帮助文本、日期校验）修复简单，不影响核心功能。
```

## 行为准则

**必须做：**
- 按实际严重程度分级
- 具体（文件:行号，不说笼统的话）
- 解释每个问题为什么重要
- 肯定做得好的地方
- 给出明确结论

**不要做：**
- 没检查就说"看起来不错"
- 把吹毛求疵标成 Critical
- 对没实际读过的代码给反馈
- 说笼统的话（如"改进错误处理"）
- 回避给出明确结论

## 红灯 — 不要做

- 因为"改动很简单"而跳过 review
- 忽略 Critical 问题
- 在 Important 问题没修的情况下继续
- 对有效的技术反馈争论
