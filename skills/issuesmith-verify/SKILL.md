# 完成前验证

## 概述

声称工作完成而不验证，不是效率问题，是不诚实。

**核心原则：** 证据先于声称，始终如此。

**违反这些规则的字面含义，就是违反其精神。**

## 铁律

```
没有新鲜的验证证据不声称完成
```

这次对话里没跑验证命令，就无权声称它通过。

## 验证关卡

```
在声称任何状态或表达满意之前：

1. 确认：什么命令能证明这个声称？
2. 运行：执行完整命令（全新、完整运行）
3. 读取：完整输出，检查退出码，数失败数
4. 验证：输出是否证实了你的声称？
   - 如果否：用证据说明实际状态
   - 如果是：带着证据做出声称
5. 确认之后：再做出声称

跳过任何一步 = 糊弄，不是验证
```

## 常见验证失败

| 声称 | 需要 | 不充分 |
|------|------|--------|
| 测试通过 | 测试命令输出：0 个失败 | 上一次跑的结果、"应该能过" |
| Lint 干净 | Lint 输出：0 个错误 | 部分检查、推断 |
| 构建成功 | 构建命令：exit 0 | Lint 通过、日志看着没问题 |
| Bug 修复 | 复现 bug 的测试通过 | 代码改了、假设修好了 |
| 回归测试生效 | 红-绿 循环验证通过 | 测试只通过一次 |
| 子任务完成 | VCS diff 显示变更 | 子任务报告"成功" |
| 需求满足 | 逐条核对 checklist | 测试通过 |

## 红灯 — 停止

- 用"应该"、"大概"、"看起来"
- 在验证之前表达满意（"好了！"、"搞定！"、"没问题！"）
- 即将提交/推送/创建 PR 但还没验证
- 信任子任务的成功报告
- 依赖部分验证
- 心想"就这一次"
- **任何暗示成功但没跑验证的措辞**

## 常见自我合理化

| 借口 | 真相 |
|------|------|
| "应该能过了" | 运行验证。"应该" ≠ "确实"。 |
| "我有信心" | 信心 ≠ 证据。 |
| "就这一次" | 没有例外。 |
| "Lint 过了" | Lint ≠ 编译。 |
| "子任务说成功了" | 独立验证。 |
| "我累了" | 疲惫 ≠ 借口。 |
| "部分检查够了" | 部分检查什么都证明不了。 |
| "换了个说法就不算声称完成" | 精神大于字面。 |

## 模式速查

**测试：**
```
✅ [运行测试命令] [看到: 34/34 pass] "全部测试通过"
❌ "应该能过" / "看着没问题"
```

**回归测试（TDD 红-绿）：**
```
✅ 写测试 → 跑（通过）→ 撤销修复 → 跑（必须失败）→ 恢复 → 跑（通过）
❌ "我写了个回归测试"（没有红-绿验证）
```

**构建：**
```
✅ [运行构建] [看到: exit 0] "构建通过"
❌ "Lint 过了"（lint 不检查编译）
```

**需求：**
```
✅ 重读 plan → 创建 checklist → 逐条验证 → 报告缺口或完成
❌ "测试通过，阶段完成"
```

**子任务委派：**
```
✅ 子任务报告成功 → 检查 VCS diff → 验证变更 → 报告实际状态
❌ 信任子任务报告
```

## 何时执行

**在以下操作之前必须执行：**
- 任何形式的成功/完成声称
- 任何形式的满意表达
- 任何关于工作状态的正面陈述
- 提交、创建 PR、标记任务完成
- 进入下一个任务
- 委派给子任务

**规则适用于：**
- 精确的措辞
- 同义词和转述
- 暗示成功
- 任何传达完成/正确的沟通

## 底线

**验证没有捷径。**

运行命令。读取输出。然后再声称结果。

这不可协商。
