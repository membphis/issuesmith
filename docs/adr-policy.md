# ADR 策略（Architecture Decision Records）

本文档说明在 IssueSmith 工作流中，什么时候需要创建 ADR，什么时候不需要，以及 ADR 的格式规范。

## 什么是 ADR

ADR（Architecture Decision Record）是一份轻量文档，用于记录重要的架构决策、上下文背景、考虑的替代方案以及最终选择的理由。它帮助项目维护者和 AI 工具理解"为什么代码是现在这个样子"。

**关键原则**：ADR 不是文档的替代品，也不是每做一个决定就要写一个。它的价值在于记录非显而易见的决策。

## 何时需要 ADR

以下场景**应该**创建 ADR：

### 1. 重要技术选型

选择对整个项目有深远影响的工具或框架时：

- 编程语言或框架的重大版本升级。
- 选择数据库、消息队列、缓存方案。
- 引入新的第三方 SDK 或 CLI 工具。

### 2. 架构模式变更

改变系统结构或模块边界时：

- 从单体拆分为多个服务。
- 引入新的设计模式（如事件驱动、CQRS）。
- 修改关键模块之间的协议或契约。

### 3. 引入或替换核心依赖

- 添加新的核心依赖（不仅仅是 dev 工具或测试库）。
- 替换现有的关键依赖（如从 Redis 迁移到 Valkey）。
- 移除重要的已有能力。

### 4. 采用非显而易见的方案

如果一个问题有多种合理方案，你选择了其中一个，而其他方案看起来也合理：

- 选择方案 A 而非方案 B 的原因是什么？
- 哪些 trade-off 被接受？
- 有哪些可以量化的考量（性能、复杂度、维护成本）？

### 5. 标记废弃或移除重要功能

- 决定停止维护某功能。
- 定义 deprecation 时间线和迁移路径。
- 移除被广泛使用或曾经重要的组件。

## 何时不需要 ADR

以下场景**不需要**创建 ADR：

- **日常 bug fix**：修复一个已知 bug，且不涉及架构变更。
- **小范围重构**：重命名变量、提取方法、整理目录结构等，不改变行为。
- **配置调整**：修改环境变量、CI 脚本、lint 规则等。
- **已有 ADR 范围内的补充实现**：如果决策已被现有 ADR 覆盖，不需要新建。
- **明确的性能优化**：使用标准优化手段，没有架构层面的选择。
- **纯文档更新**：更新 README、注释或其他说明性文档。
- **测试补充**：添加新的测试用例来覆盖已有功能。

**一个简单的判断标准**：如果六个月后的你自己（或 AI 工具）问"为什么当初这么设计？"，这个答案目前是否显而易见？如果不是——创建 ADR。

## ADR 存放位置

所有 ADR 存放在 `docs/adr/` 目录下，按序号命名：

```
docs/adr/
├── 0001-record-architecture-decisions.md
├── 0002-use-sqlite-for-local-storage.md
├── 0003-migrate-from-rest-to-graphql.md
└── ...
```

- 序号从 0001 开始，递增。
- 文件名使用短横线命名（kebab-case），简洁描述决策内容。
- 每个 ADR 一个文件，不合并多个决策。

## ADR 模板

```markdown
# ADR-NNNN: [决策标题]

## 状态

[Proposed / Accepted / Deprecated / Superseded by ADR-NNNN]

## 日期

YYYY-MM-DD

## 背景

描述要解决的架构问题、触发这个决策的上下文，以及相关的约束条件。

## 决策

一句话说明最终选择是什么。

## 考虑的方案

### 方案 A: [描述]

- 优点：
- 缺点：

### 方案 B: [描述]

- 优点：
- 缺点：

## 影响

- [ ] 需要更新哪些文档？
- [ ] 需要迁移哪些已有代码？
- [ ] 对下游消费者有什么影响？
- [ ] 是否包含 breaking change？

## 参与讨论

- @参与者 1
- @参与者 2
```

## ADR 生命周期

```
Proposed → Accepted → [Deprecated / Superseded]
```

1. **Proposed**：ADR 在 PR 中被提出，等待讨论和审查。
2. **Accepted**：决策被接受并合并到主分支，开始执行。
3. **Deprecated**：决策不再适用（但未完全被替代方案取代）。
4. **Superseded**：被新的 ADR 取代，记录新的 ADR 编号。

只维护**当前有效**的 ADR。被取代的 ADR 保留在仓库中以供历史参考，但应更新状态为 Deprecated 或 Superseded。

## ADR 编写规范

- **简洁**。一份 ADR 通常在 1-2 页之间，应让读者在 5 分钟内理解决策。
- **客观**。列出考虑过的所有方案及其真实的 pros/cons，不要只写"其他方案都很差"。
- **独立**。每个 ADR 记录一个独立决策。如果需要关联其他 ADR，使用引用。
- **面向未来读者**。把自己当成六个月后的新人，用清晰的文字解释当下的选择。

## 与 Issue 的关系

| 信息来源 | ADR |
|----------|-----|
| Issue 中的讨论 | 提炼为 ADR 的"考虑的方案" |
| Issue 的 Notes/Decisions | 作为 ADR 的决策理由 |
| PR review 中的讨论 | 补充到 ADR 的"参与讨论"字段 |

通常流程是：Issue 讨论 → 做出决策 → 创建 ADR 记录 → PR 中包含 ADR → 合并后 ADR 生效。
